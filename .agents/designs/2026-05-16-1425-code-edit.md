# DESIGN: Code Editing Operations

## Status

Initial generic file editing exists through `file_edit`. Language-aware editing
is intentionally out of scope for the current project and Go language roadmap.
Explicit `go_fmt` formatting is tracked in the Go plugin roadmap because it is
a named Go toolchain operation, not an abstract edit/refactor primitive.

## Summary

Code editing is an abstract runtime capability. It should not be treated as a
Go-plugin feature, even when Go-aware operations eventually use Go parsers,
`go/packages`, or `go/types` under the hood.

The split is:

- `file_edit`: generic existing-file mutation by explicit coordinates.
- future language edit operations: semantic edits that compute file changes
  from language-aware inputs.
- Go-specific helpers: implementation details used by language edit
  operations, not the public shape of code editing as a whole.

## Current Generic Operation

`file_edit` remains file-oriented:

- it edits existing files only;
- it uses original-file coordinate semantics;
- it merges non-overlapping atomic edits;
- it supports dry-run output;
- it can render full, atomic, or no diff output.

Do not put AST editing into `file_edit`. Its contract is intentionally
language-neutral.

## Future Language-Aware Editing

Language-aware mutation should be exposed through separate operations with
semantic inputs, not line-number patch scripts. Candidate Go-backed operations
include:

- `go_update_imports`
- `go_replace_function_body`
- `go_add_function`
- `go_add_method`
- `go_rename_symbol`

These are examples of language edit operations, not evidence that code editing
belongs inside the Go project roadmap. Equivalent operations for other
languages should share the same abstract expectations.

## Contract Requirements

Language-aware edit operations should:

- resolve against current source structure before computing changes;
- produce a dry-run diff by default;
- route final writes through `runtime/system.Workspace`;
- use `file_edit` or the same atomic edit machinery for final file mutation;
- report the symbols, files, and ranges that drove the edit;
- include diagnostics when source parsing, type loading, or edit application is
  incomplete;
- avoid hidden formatting unless formatting is explicitly part of the
  operation contract.

Explicit formatting commands such as `go_fmt` may live in a language plugin
when they are advertised as side-effecting toolchain operations. They should
not be invoked implicitly by generic editing or refactoring operations.

## Go-Aware Implementation Direction

Go edit operations may use:

- `go/parser`, `go/ast`, and `go/token` for syntax-local edits;
- `go/packages` and `go/types` where type identity or cross-file resolution is
  required;
- `gofmt`/`go/format` only when the operation explicitly owns formatting.

Recommended rollout:

1. `go_update_imports`
2. `go_replace_function_body`
3. `go_add_function`
4. `go_add_method`
5. `go_rename_symbol`

`go_rename_symbol` comes last because cross-file rename needs the highest
correctness bar and should be type-aware before it is advertised as reliable.

## Non-Goals

For the current Go project/language roadmap, do not add code editing milestones
except as dependencies or consumers of Go read/navigation capabilities. Keep
the roadmap focused on project inventory, Go structural reads, navigation,
documentation, and process-backed Go commands.
