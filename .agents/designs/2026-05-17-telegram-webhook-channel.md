# DESIGN: Telegram Webhook Channel Provider

## Status

Draft.

## Summary

Add a Telegram channel provider that receives Telegram Bot API updates through a
webhook, normalizes them into `core/channel.Inbound`, sends them through
`orchestration/harness`, and replies through Telegram Bot API methods. In local
development, the provider can use the generic tunnel design to expose the local
webhook listener through Cloudflare Quick Tunnel or ngrok.

Production mode uses a configured public HTTPS URL. Development mode can derive
that URL from a tunnel attached to the local HTTP listener.

## Goals

- Provide a secure inbound Telegram channel using Bot API webhooks.
- Reuse generic tunnel providers for local development.
- Keep Telegram protocol/wire types in adapters, not core.
- Preserve the existing channel-to-session path through `orchestration/harness`.
- Support conservative trust mapping and explicit chat/user allowlists.
- Provide basic outbound text replies with room for richer operations later.

## Non-goals

- Implementing Telegram MTProto or userbot behavior.
- Making Telegram a trusted operator surface by default.
- Implementing payments, moderation, Mini Apps, or files in the MVP.
- Adding Telegram-specific fields to `core/channel`.

## Telegram Receive Modes

Telegram Bot API supports two practical inbound modes:

- webhook: Telegram sends HTTPS `POST` requests to a configured public URL.
- long polling: the bot calls `getUpdates` over outbound HTTPS.

This design focuses on webhook mode. Long polling can be a later development
fallback, but webhook mode is the production shape and the best fit for daemon
channels.

## Relationship To Generic Tunnels

The Telegram adapter needs a public webhook URL. In production this should be a
stable configured URL:

```yaml
daemon:
  channels:
    - name: telegram-prod
      type: telegram
      connector: telegram-main
      listener: webhook-http
      webhook:
        public_url: https://bot.example.com/telegram/webhook/${nonce}
```

In local development it can be derived from a tunneled listener:

```yaml
daemon:
  listeners:
    - name: webhook-http
      type: http
      addr: 127.0.0.1:8080
      tunnel:
        provider: cloudflare
        mode: quick
        enabled: true

  channels:
    - name: telegram-dev
      type: telegram
      connector: telegram-main
      listener: webhook-http
      webhook:
        path: /telegram/webhook/${nonce}
        register: on_start
```

Daemon startup order:

1. Start local HTTP listener.
2. Start configured tunnel for that listener.
3. Wait for tunnel public URL.
4. Compose Telegram webhook URL from public base URL + channel path.
5. Call Telegram `setWebhook` with `secret_token` and `allowed_updates`.
6. Begin receiving webhook updates.

The Telegram provider should not start Cloudflare or ngrok directly. It consumes
the listener's public URL from daemon/tunnel orchestration.

## Config Shape

Draft connector config:

```yaml
connectors:
  - name: telegram-main
    type: telegram
    token_ref: env:TELEGRAM_BOT_TOKEN
    webhook_secret_ref: env:TELEGRAM_WEBHOOK_SECRET
```

Draft channel config:

```yaml
daemon:
  channels:
    - name: telegram-dev
      type: telegram
      connector: telegram-main
      listener: webhook-http
      session: support
      webhook:
        path: /telegram/webhook/${nonce}
        public_url: ""          # optional override
        register: on_start      # never | on_start | manual
        delete_on_stop: true
        allowed_updates:
          - message
          - edited_message
          - callback_query
          - my_chat_member
      access:
        mode: restricted
        allow_chats:
          - "-1001234567890"
        allow_users:
          - "12345678"
        allow_chat_types:
          - private
          - group
          - supergroup
        default_trust: untrusted
        verified_trust_for_allowlist: true
      threading:
        mode: chat_topic
```

Notes:

- `token_ref` and `webhook_secret_ref` are secret refs, not inline values.
- `path` should include a random nonce or be generated by config loading.
- `public_url` can override the tunneled listener public URL.
- `register: manual` lets an operator use `cloud_tunnel` and Telegram webhook
  operations explicitly.

## Webhook Registration

Use Telegram `setWebhook` with:

- `url`: public HTTPS webhook URL.
- `secret_token`: generated or configured secret.
- `allowed_updates`: smallest update set required by the channel.
- optional `drop_pending_updates` in development when explicitly configured.

Use `deleteWebhook` on stop only when configured. For shared production bots,
blindly deleting a webhook on daemon exit can interrupt service.

Webhook registration is side-effecting and should be visible as an operation or
lifecycle event.

## Inbound HTTP Security

For each webhook request:

1. Require `POST`.
2. Require JSON body and enforce a small maximum body size.
3. Verify `X-Telegram-Bot-Api-Secret-Token` with constant-time comparison.
4. Decode one Telegram `Update`.
5. Deduplicate by `update_id` where feasible.
6. Reject or ignore update types outside `allowed_updates`.
7. Apply channel access policy before creating executable inbound input.
8. Return quickly. Long-running agent work should not hold the Telegram request
   longer than necessary.

Recommended responses:

- `401` or `403` when the secret token is missing or wrong.
- `400` for malformed JSON.
- `200` for intentionally ignored Telegram updates to avoid retry storms.
- `202` or `200` when accepted for processing.

## Trust And Identity

Webhook secret verification proves that the update was delivered through the
configured Telegram webhook. It does not prove that the Telegram user is trusted.

Default mapping:

```go
policy.Caller{
    Kind:   policy.CallerUser,
    Source: "telegram",
    Principal: policy.Principal{
        Kind: "telegram_user",
        ID:   from.ID,
        Name: displayName,
    },
}

policy.Trust{
    Kind:       policy.TrustInvocation,
    Level:      policy.TrustUntrusted,
    VerifiedBy: "telegram.webhook.secret",
}
```

If `allow_users` or `allow_chats` matches and the config opts in, the adapter may
raise to `TrustVerified`. It should never assign `TrustPrivileged` or
`TrustSystem` from Telegram input alone.

Group and channel messages should be especially conservative. By default, only
respond in groups when one of these is true:

- the bot is mentioned,
- the message is a reply to the bot,
- the message is a bot command addressed to the bot,
- the chat is explicitly allowlisted for ambient messages.

## Conversation Mapping

Telegram conversation IDs should be stable and protocol namespaced:

- private chat: `telegram:private:<user_id>`
- chat/group/supergroup: `telegram:chat:<chat_id>`
- forum topic: `telegram:chat:<chat_id>:thread:<message_thread_id>`
- channel post: `telegram:channel:<chat_id>`
- callback query: reuse the message chat/thread when available, otherwise
  `telegram:callback:<callback_query_id>`

Threading modes:

```yaml
threading:
  mode: private_user | chat | chat_topic | message
```

Default should be `chat_topic` for groups/supergroups and `private_user` for
private chats.

## Update Normalization

MVP update handling:

### `message`

- Text messages become `channel.InboundMessage`.
- Telegram message metadata is preserved under `Message.Metadata`.
- Bot commands are preserved as metadata and may be converted to
  `channel.InboundCommand` only through existing command parsing boundaries.

### `edited_message`

- For MVP, either ignore with `200` or turn into a message with metadata
  `telegram.edited=true`.
- Later, edited messages can become an event.

### `callback_query`

- Convert to `channel.InboundMessage` or `channel.InboundCommand` depending on
  callback payload shape.
- Always call `answerCallbackQuery` quickly when configured, so Telegram clients
  stop showing loading state.

### `my_chat_member`

- Use as a lifecycle/access event: bot added, removed, blocked, promoted.
- MVP may log/emit event without session execution.

## Outbound Mapping

MVP:

- `channel.Outbound{Kind: message}` -> `sendMessage`.
- Preserve chat ID and optional thread ID from inbound metadata/session routing.
- Reply to source message when configured.
- Chunk long text to Telegram message limits.
- Use plain text by default.

Later:

- MarkdownV2/HTML rendering with escaping.
- `editMessageText` for progress updates.
- inline keyboards for approvals.
- file/photo/document sends.

## Operations

Telegram-specific operations can be contributed by a Telegram plugin. Keep the
first set small:

```text
telegram.set_webhook
telegram.delete_webhook
telegram.get_webhook_info
telegram.send_message
telegram.answer_callback_query
```

Future operations:

```text
telegram.edit_message_text
telegram.send_document
telegram.get_file
telegram.download_file
telegram.set_my_commands
telegram.get_chat
telegram.get_chat_member
telegram.approve_chat_join_request
telegram.restrict_chat_member
telegram.answer_inline_query
telegram.send_poll
```

The generic tunnel operation remains separate:

```text
cloud_tunnel op=create provider=cloudflare local_url=http://127.0.0.1:8080
```

A development workflow can compose both:

1. `cloud_tunnel create` starts Cloudflare Quick Tunnel.
2. `telegram.set_webhook` registers the tunnel URL.
3. Telegram webhook updates enter the channel.

## Architecture Placement

- `adapters/telegram`:
  - Telegram Bot API client.
  - webhook HTTP handler.
  - Telegram update DTOs.
  - normalization to `channel.Inbound`.
  - outbound `sendMessage` mapping.
- `plugins/telegram`:
  - channel provider contribution.
  - Telegram operations.
  - optional config/resource contribution.
- `orchestration`:
  - daemon/channel assembly and public URL handoff from listener tunnels.
  - calls into `harness.HandleInbound` only through normalized channel input.
- `core/channel`:
  - unchanged.
- `core` config later:
  - only inert stable channel/tunnel specs if the app config shape is promoted.

Flow:

```text
Telegram HTTPS webhook
  -> adapters/telegram webhook handler
  -> channel.Inbound
  -> orchestration/harness.HandleInbound
  -> session execution
  -> channel.Outbound
  -> adapters/telegram sendMessage
```

Development tunnel flow:

```text
cloudflared/ngrok managed process
  -> public HTTPS URL
  -> daemon listener public URL
  -> Telegram setWebhook URL
  -> webhook flow above
```

## Security

- Never log bot token, webhook secret, or full Bot API URLs containing tokens.
- Use constant-time comparison for webhook secret token.
- Use random route nonce in webhook path.
- Keep body size and execution time bounded.
- Ignore bot/self messages to avoid loops.
- Rate limit per chat and per user.
- Default to untrusted/public authority.
- Require allowlists or approval for side-effecting operations.
- Treat Telegram usernames/display names as PII.
- Avoid full message body logging except explicit debug mode.

## Testing

Unit tests:

- webhook rejects missing/wrong secret.
- webhook accepts correct secret.
- message update normalizes to expected `channel.Inbound`.
- group message without mention is ignored by default.
- allowlisted user/chat can receive verified trust when configured.
- callback query is acknowledged and normalized.
- outbound message calls expected Bot API endpoint with redacted errors.

Integration tests:

- fake Telegram server for `setWebhook`, `deleteWebhook`, and `sendMessage`.
- fake tunnel provider returning a public URL.
- daemon startup composes tunnel public URL and webhook path correctly.

Live/manual dev test:

1. Start local daemon with Cloudflare Quick Tunnel.
2. Confirm public URL appears.
3. Confirm Telegram webhook is registered.
4. Send bot a private message.
5. Confirm `channel.Inbound` reaches harness and bot replies.

## Implementation Sequence

1. Implement generic tunnel provider draft and Cloudflare Quick Tunnel support.
2. Add public URL handoff from tunneled listener to channel assembly.
3. Add Telegram connector config and Bot API client.
4. Add Telegram webhook handler with secret validation.
5. Add update normalization for `message` and `callback_query`.
6. Add `sendMessage` outbound path.
7. Add webhook registration lifecycle using listener public URL.
8. Add docs for Cloudflare Quick Tunnel development setup.
9. Add optional ngrok support through the generic tunnel provider model.

## Open Questions

- Should long polling be supported as a first-class dev mode, or should local dev
  use tunnels exclusively?
- Should Telegram webhook registration be automatic by default in development
  only?
- How should pending updates be handled when switching between tunnel URLs?
- Should `callback_query` payloads be parsed as commands by default or remain
  messages/events until a command binding opts in?
- Should channel-level tunnel config be syntactic sugar for listener-level tunnel
  config, or should channels be allowed to own dedicated tunnel processes?
